# Create Issue Command

このコマンドは対話的な議論を通じてGitHub Issueを効率的に作成します。

**重要**: 全てのルール・規則は [Issue作成ガイド](../docs/PROJECT_MANAGEMENT/ISSUE_CREATION_GUIDE.md) に従うこと。

## 基本方針

- **対話ベース**: コマンド実行後にタスク内容を議論
- **要件整理**: 何をIssueにするか一緒に決定
- **適切な分類**: 議論を通じて最適なラベル・優先度を決定
- **親子関係**: 必要に応じて親チケット・子チケット構造を提案

## 使用方法

```
/create-issue
```

**引数なし**: 対話的にIssue内容を決定

## 実行フロー

### 1. 要件ヒアリング
- 何を実現したいかの議論
- 背景・目的の整理
- 技術的制約・要件の確認

### 2. Issue構造の提案
- 単一Issue vs 親子チケット構造の判定
- 適切な作業分割の提案
- 優先度・作業規模の相談

### 3. 内容詳細化
- タイトル・説明文の作成
- 受け入れ条件の明確化
- テンプレート選択（feature/bug/docs等）

### 4. ラベル・メタデータ決定
- 優先度（P0/P1/P2）の相談
- 作業規模（S/M/L）の見積もり
- コンポーネント・カテゴリの分類

### 5. Issue作成実行
- ISSUE_CREATION_GUIDE.md準拠でIssue作成
- 親子関係がある場合は適切にリンク設定
- 作成結果の確認・レビュー

## 対話例

```
User: /create-issue
Assistant: どのようなタスクをIssueにしたいですか？

User: ログイン機能を改善したい
Assistant: ログイン機能の改善について詳しく教えてください。
- 現在の問題点は何ですか？
- どのような改善を想定していますか？
- セキュリティ、UX、パフォーマンスのどの観点でしょうか？

[議論を通じて要件を整理し、適切なIssue構造を提案]
```

## 期待される成果

- **明確な要件定義**: 議論を通じた要件の明確化
- **適切なIssue構造**: 単一 vs 親子チケットの最適判断
- **正確な見積もり**: 作業規模・優先度の適切な設定
- **実装可能性**: 技術的制約を考慮した現実的なタスク設計
