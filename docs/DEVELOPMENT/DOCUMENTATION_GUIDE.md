# ドキュメント管理ガイド

## 📝 概要

このガイドは、プロジェクト内のドキュメント作成・管理における指針を提供します。
一貫性のあるドキュメント体系を構築し、新規開発者のオンボーディング効率と保守性を向上させることを目的としています。

## 📋 README.mdドキュメント作成時の指針

### 1. ディレクトリ別README.md作成

- **目的**: 各ディレクトリの役割・使用方針の明確化
- **対象**: src/, tests/, e2e/, public/, .github/, .vscode/ 等
- **Issue作成時のラベル**: `type-quality:docs`, `component:documentation`

### 2. ドキュメント構造・役割分担

```
ルートREADME.md     → プロジェクト概要・クイックスタート
docs/README.md      → 詳細ドキュメント一覧・ナビゲーション
各ディレクトリ/     → 具体的な技術仕様・使用方法
```

**役割分担の原則:**
- **ルートREADME.md**: 最短でのプロジェクト理解・実行開始
- **docs/**: 包括的な技術ドキュメント・詳細ガイド
- **各ディレクトリREADME.md**: 専門領域の深い技術情報

### 3. ドキュメント品質基準

- **新規開発者が迷わず理解可能**: 前提知識なしで読解できる
- **実装と一致**: 実際のコード・設定と整合性がある
- **保守しやすい粒度**: 適切な情報量・更新頻度

### 4. 推奨テンプレート構成

```markdown
# [ディレクトリ名] - [役割]

## 概要
ディレクトリの目的・役割

## ファイル構成
主要ファイルの説明

## 使用方法・設定方法
具体的な使用・設定手順

## 注意事項・ベストプラクティス
開発・保守時の注意点

## トラブルシューティング
よくある問題・解決策

## 関連ドキュメント
関連する他のドキュメント
```

## 🎯 ディレクトリ別ドキュメント方針

### src/ - ソースコード
- Vue 3 Composition API使用方針
- コンポーネント設計パターン
- TypeScript型定義ルール
- 新規コンポーネント作成指針

### tests/ - テスト
- テストファイル命名規則
- モック戦略（Supabase等）
- カバレッジ目標・測定方法
- 新規テスト作成ガイドライン

### e2e/ - E2Eテスト
- Playwright設定・環境構築
- E2Eテストシナリオ設計方針
- CI/CD連携設定
- デバッグ・トラブルシューティング

### その他ディレクトリ
- **public/**: 静的リソース管理方針
- **.github/**: CI/CDワークフロー・Issueテンプレート
- **.vscode/**: 推奨VSCode設定
- **.devcontainer/**: 開発コンテナ設定

## 📚 ドキュメント管理ベストプラクティス

### 情報の重複を避ける
- 各ドキュメントの責務を明確化
- 参照関係を適切に設定
- 一箇所での情報管理を原則とする

### 更新しやすさを重視
- 技術仕様の変更に追従しやすい構造
- 自動化可能な部分は自動化（型定義等）
- 適切な粒度での情報分割

### アクセスしやすさの確保
- 論理的な階層構造
- 明確な命名規則
- 適切なリンク・参照設定

## 🔄 ドキュメント更新フロー

### 1. 新規ドキュメント作成
```bash
# Issue作成
gh issue create --title "[Documentation] [対象]README.md作成" \
  --label "priority:P1,size:S,type-quality:docs,component:documentation"

# 作業開始
npm run start-issue [issue番号]
```

### 2. 既存ドキュメント更新
- 技術変更と同時に対応するドキュメントを更新
- PR作成時に関連ドキュメントの確認・更新
- 定期的なドキュメント棚卸し・整合性確認

### 3. 品質チェック
- 新規開発者によるドキュメント検証
- 実装との整合性確認
- 情報の過不足・重複チェック
