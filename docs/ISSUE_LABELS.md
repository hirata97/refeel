# 🏷️ ラベル管理システム

このドキュメントでは、GoalCategorizationDiaryプロジェクトで使用されているラベル体系について詳しく説明します。

## 📋 概要

効率的なIssue管理のために、以下の3つのカテゴリでラベルを体系化しています：

1. **優先度ラベル** (`priority:`) - 作業の緊急度・重要度
2. **サイズラベル** (`size:`) - 作業にかかる時間・規模  
3. **実装内容ラベル** (`type-*:`) - 作業の種類・分野

## 🚨 優先度ラベル（priority:）

| ラベル | 色 | 説明 | 使用例 |
|--------|-----|------|--------|
| `priority:P0` | 🔴 | 最高優先度（緊急・重要）| バグ修正、本番障害、リリースブロッカー |
| `priority:P1` | 🟡 | 高優先度（重要）| 主要機能追加、パフォーマンス改善 |
| `priority:P2` | 🔵 | 中優先度（通常）| 機能改善、リファクタリング、ドキュメント |

### 優先度の決定基準

**P0（最高優先度）**
- 本番環境に影響するバグ
- セキュリティ脆弱性
- リリースを阻害する問題
- ユーザー体験を著しく損なう問題

**P1（高優先度）**
- 新機能開発のメイン機能
- パフォーマンスの大幅改善
- 重要なリファクタリング

**P2（中優先度）**
- UI/UXの改善
- 小規模な機能追加
- ドキュメント整備
- テストカバレッジ向上

## 📏 サイズラベル（size:）

| ラベル | 色 | 説明 | 目安時間 | 例 |
|--------|-----|------|----------|-----|
| `size:S` | 🔴 | 小規模 | 1-2日 | バグ修正、設定変更、小さなUI調整 |
| `size:M` | 🟡 | 中規模 | 3-5日 | 新機能追加、コンポーネント作成 |
| `size:L` | 🔵 | 大規模 | 1週間以上 | 大型機能開発、アーキテクチャ変更、CI/CD構築 |

### サイズ見積もりのガイドライン

**S（小規模）**
- 単一ファイルの修正
- 設定ファイルの変更
- 簡単なバグ修正
- 小さなUI調整

**M（中規模）**
- 複数ファイルにまたがる変更
- 新しいコンポーネントの作成
- 既存機能の拡張
- テストの追加

**L（大規模）**
- 新しい主要機能の開発
- データベーススキーマの変更
- アーキテクチャの大幅な変更
- CI/CDパイプラインの構築

## 🔧 実装内容ラベル（type-*:）

### type-basic: 基本的な作業タイプ

| ラベル | 色 | 説明 | 例 |
|--------|-----|------|-----|
| `type-basic:bugfix` | 🔴 | バグ修正 | 機能不具合、エラー修正 |
| `type-basic:enhancement` | 🟠 | 既存機能改善 | UI/UX改善、ユーザビリティ向上 |
| `type-basic:feature` | 🟢 | 新機能追加 | 新しい機能・ページの実装 |
| `type-basic:refactor` | 🟣 | リファクタリング | コード整理、構造改善 |

### type-infra: インフラ・技術タイプ

| ラベル | 色 | 説明 | 例 |
|--------|-----|------|-----|
| `type-infra:automation` | 🟠 | 自動化・スクリプト | ビルドスクリプト、自動化ツール |
| `type-infra:ci-cd` | 🔵 | CI/CD・パイプライン | GitHub Actions、デプロイ設定 |
| `type-infra:performance` | 🔵 | パフォーマンス改善 | 速度最適化、メモリ効率化 |
| `type-infra:security` | 🔴 | セキュリティ | 脆弱性修正、認証・認可強化 |

### type-quality: 品質・ドキュメントタイプ

| ラベル | 色 | 説明 | 例 |
|--------|-----|------|-----|
| `type-quality:docs` | 🔵 | ドキュメント | README更新、コメント追加、仕様書作成 |
| `type-quality:test` | 🟡 | テスト関連 | ユニットテスト、E2Eテスト、テストカバレッジ |

### 特殊ラベル（z-）

| ラベル | 説明 | 使用場面 |
|--------|------|----------|
| `z-good-first-issue` | 初心者向け | 新規コントリビューター向けの簡単なタスク |
| `z-help-wanted` | コミュニティ協力歓迎 | 外部からの協力を求めるタスク |

## 🤖 自動ラベリング機能

### 自動付与される条件

GitHub ActionsによりIssue作成時に以下の条件で自動ラベリングされます：

#### type-basicラベル
- `bug`、`バグ` → `type-basic:bugfix`
- `feature`、`機能` → `type-basic:feature`
- `enhancement`、`改善` → `type-basic:enhancement`  
- `refactor`、`リファクタ`、`クリーンアップ` → `type-basic:refactor`

#### type-infraラベル
- `CI/CD`、`パイプライン` → `type-infra:ci-cd`
- `automation`、`自動化` → `type-infra:automation`
- `security`、`セキュリティ` → `type-infra:security`
- `performance`、`パフォーマンス` → `type-infra:performance`

#### type-qualityラベル
- `test`、`テスト` → `type-quality:test`
- `docs`、`documentation`、`ドキュメント` → `type-quality:docs`

#### 優先度ラベル
- `緊急`、`urgent`、`critical` → `priority:P0`

### 自動アサイン機能

- Issue作成者が自動的にアサインされます
- 作業開始時の手動アサインが不要になります

## 📝 ラベル使用のベストプラクティス

### Issue作成時

1. **必須ラベル**を必ず付与：
   - 優先度ラベル（`priority:P0`〜`P2`）
   - サイズラベル（`size:S`〜`L`）
   - 実装内容ラベル（`type-*:`）

2. **自動ラベリングを活用**：
   - タイトルに適切なキーワードを含める
   - 日本語・英語どちらでもOK

### ラベル管理時

1. **一貫性を保つ**：
   - 同じような作業には同じラベルを使用
   - チーム内でラベルの意味を共有

2. **定期的な見直し**：
   - プロジェクトの進行に応じてラベル体系を調整
   - 不要になったラベルは削除
